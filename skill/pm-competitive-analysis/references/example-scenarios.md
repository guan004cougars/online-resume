# 示例场景

这个文件用于帮助识别：什么样的用户请求适合触发“产品经理竞品分析”这个 skill。

## 一、典型触发场景

以下场景通常应该触发这个 skill：
- 用户希望分析某个赛道的竞品格局
- 用户希望看清某个市场的竞争格局和玩家分布
- 用户希望比较几个产品的差异
- 用户希望从产品经理视角拆解竞品
- 用户希望判断某个行业真正的竞争要素是什么
- 用户希望梳理竞品所属企业的发展路径、产品关键路径和关键转折点
- 用户希望用数据佐证竞品判断
- 用户希望看清竞品的核心壁垒、盈利模式和收入结构
- 用户希望为立项、功能规划、增长、定价提供参考
- 用户希望输出一份带结论、建议和可视化展示的竞品分析
- 用户希望看到二维四象限、市场份额分布和头中尾分层

如果用户只给了一个非常宽泛的赛道名称，也应触发这个 skill，但第一步不是直接分析，而是先帮助用户收敛赛道范围。

## 二、典型用户提问示例

### 1. 立项分析类

适合触发：
- 帮我分析一下 AI 笔记产品的竞品格局
- 我想做一个面向职场人的效率工具，先帮我看一下现在市面上的主要竞品
- 帮我做一下这个赛道的竞品分析，看看还有没有机会切入
- 请从产品经理角度分析这个方向值不值得做
- 我想判断这个市场是不是已经太拥挤了

这类请求的重点：
- 市场关键竞争要素
- 市场格局
- 市场竞争格局
- 核心壁垒
- 主要玩家分层
- 机会点与空白点
- 是否值得进入

### 0. 赛道过大需要先收敛类

适合触发：
- 帮我分析一下医疗赛道
- 想看教育行业的竞品
- 电商赛道有哪些竞品值得看
- 帮我做一下 AI 赛道调研

这类请求的正确处理方式：
- 先说明赛道过大，直接分析会失焦
- 先拆成几个细分方向供用户选择
- 如果用户没有目标用户，再补目标用户候选
- 如果用户没有分析目标，再补分析目标候选
- 再基于细分方向生成核心场景候选
- 再进入正式竞品分析

这类请求的重点：
- 赛道收敛
- 细分方向选择
- 目标用户候选
- 分析目标候选
- 核心场景候选

### 2. 功能规划类

适合触发：
- 帮我比较一下 Notion、飞书文档、语雀在知识库功能上的差异
- 我们想做一个 AI 搜索功能，帮我拆解几家竞品的设计方式
- 请对比一下这几个产品在核心流程上的优缺点
- 帮我看看别人是怎么做新手引导的
- 我想知道这个功能哪些是基础能力，哪些是差异化能力

这类请求的重点：
- 该功能背后的关键竞争要素
- 功能设计
- 关键流程
- 体验差异
- 对我方功能规划的启发

### 3. 定价与商业模式类

适合触发：
- 帮我分析一下这几个产品的定价策略
- 对比一下它们的会员体系和付费逻辑
- 这些竞品是怎么把免费用户转成付费用户的
- 帮我拆一下它们的收费边界设计
- 我想看一下不同竞品的商业模式差异
- 帮我看看它们各自到底靠什么挣钱
- 我想知道这几家公司的收入主要来自哪里

这类请求的重点：
- 价格竞争是否是核心竞争要素
- 套餐结构
- 付费点设计
- 免费与付费边界
- 商业模式逻辑
- 盈利模式
- 收入结构

### 3.1 壁垒与护城河类

适合触发：
- 帮我分析一下这家产品到底有什么核心壁垒
- 它为什么能做成，别人为什么不容易复制
- 请拆一下它的护城河和增长飞轮
- 我想知道这家公司到底是不是有真壁垒

这类请求的重点：
- 解决了什么痛点
- 是否刚需
- 壁垒来源
- 为什么只有它更有机会做成
- 壁垒和商业模式是否互相支撑

### 4. 增长策略类

适合触发：
- 帮我分析一下这些竞品是怎么获客的
- 我想看一下几家产品的增长打法有什么不同
- 请从拉新、转化、留存角度比较一下这几个产品
- 帮我拆一下竞品的内容策略和传播方式
- 我想参考一下它们的增长路径

这类请求的重点：
- 流量渠道是不是核心竞争要素
- 获客渠道
- 转化路径
- 首次价值体验
- 品牌与传播方式

### 5. 汇报总结类

适合触发：
- 帮我整理一份给老板看的竞品分析
- 请做一个结论先行的竞品对比总结
- 我需要一份可以直接汇报的市场格局分析
- 帮我快速总结这几个竞品的关键差异和建议
- 请用产品经理视角输出一版竞品结论

这类请求的重点：
- 结论清晰
- 关键数据可引用
- 信息压缩
- 对我方建议明确
- 适合快速阅读和汇报

### 6. 企业路径与战略变化类

适合触发：
- 帮我梳理一下这个产品背后企业的关键战略变化
- 我想知道这款产品为什么会从工具转向平台
- 请按时间轴拆解这个产品的重要迭代和企业转折点
- 帮我看一下出海策略对这个产品形态有什么影响
- 我想知道这几家竞品的企业路线差异
- 帮我拆一下这个产品每个阶段为什么这么做

这类请求的重点：
- 企业/产品关键路径
- 关键转折点
- 战略动作对产品的影响
- 阶段主题
- 决策逻辑
- 时间轴展示

### 7. 数据佐证与展示类

适合触发：
- 帮我用数据支持这份竞品分析
- 请补一下这几个产品的 MAU、下载量、市场份额
- 帮我做一个市场竞争格局图，看看谁在什么位置
- 做成一份可以本地打开的 HTML 竞品报告
- 我不要纯文本，帮我做成图表和卡片展示
- 请把数据来源也一起标上

这类请求的重点：
- 关键数据指标
- 数据来源可信度
- 二维四象限
- 市场份额分布
- 头中尾分层
- 核心壁垒
- 盈利模式
- 收入结构
- HTML 展示
- 图表、卡片、时间轴

## 三、应当触发但用户表达不标准的情况

有些用户不会直接说“竞品分析”，但本质上是同一类任务，这时也应考虑触发。

例如：
- 帮我看看别人是怎么做这个的
- 这个方向有哪些做得比较好的产品
- 你觉得我们应该参考哪几家
- 这些产品的差异在哪里
- 用户为什么会选 A 不选 B
- 这几个产品到底谁更适合哪类人
- 这个市场到底靠什么竞争
- 这家产品为什么这两年变化这么大
- 能不能做成一个本地可以预览的报告

遇到这类请求时，应主动转化为竞品分析任务来处理。

## 四、不应触发的情况

以下场景通常不需要触发这个 skill：
- 单纯介绍某一个公司的基本信息
- 纯行业新闻汇总
- 没有比较目标的百科式介绍
- 只要求翻译网页内容
- 纯技术实现对比，没有产品决策语境
- 学术论文综述或研究文献整理

如果用户只是想知道“这个产品是什么”，而不是“它和谁比、差异在哪、对我方意味着什么”，则不必触发。

## 五、触发后的默认处理方式

一旦判断适合触发，应默认按以下思路推进：

1. 先判断赛道是否过大
2. 再判断是否需要先生成目标用户候选
3. 再判断是否需要先生成分析目标候选
4. 再判断是否需要先生成核心场景候选
5. 再定义竞品范围
6. 识别市场关键竞争要素
7. 绘制市场竞争格局
8. 收集关键数据与证据
9. 对核心竞品做统一拆解
10. 做横向比较
11. 提炼洞察
12. 输出建议

不要一上来就堆功能清单。

## 六、默认假设处理

如果用户信息不足，可以做这些合理假设：
- 只给了大赛道：优先先拆分细分方向，而不是直接假设一个具体市场
- 已有细分赛道，但缺目标用户：优先由模型生成 3-5 个目标用户候选
- 已有细分赛道和目标用户，但缺分析目标：优先由模型生成常见分析目标候选
- 已有细分赛道、目标用户、分析目标，但缺核心场景：优先由模型生成 3-5 个核心场景候选，而不是让用户自由发挥
- 没写分析目标：优先根据任务语气推断是立项、功能规划、增长还是汇报
- 没给竞品名单：先补一个合理的初始竞品集合
- 没说输出格式：默认使用“分析目标 + 竞品分组 + 市场关键竞争要素 + 核心拆解 + 洞察 + 建议”
- 没说深度：默认先给中等深度、结论导向的版本
- 没说呈现方式：在适合汇报或展示时，优先考虑 HTML 报告

## 七、输出风格示例

更好的表达方式：
- 这几家产品表面上功能接近，但核心竞争方式不同
- A 更偏效率工具，B 更偏协作平台，C 更偏内容沉淀
- 这个能力值得跟进，但不建议照搬其交互方式
- 目前市场不是没有机会，而是需要换一个切入口
- 用户选择竞品的关键不是功能多少，而是完成核心任务的效率与信任感

不够好的表达方式：
- 功能很多，体验很好，值得学习
- 各家都有自己的优势
- 市场竞争激烈，需要持续关注
- 建议结合自身情况进行优化

应避免空泛结论，尽量形成明确判断。
