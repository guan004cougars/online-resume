# 产品经理竞品分析

当用户提出以下类型的需求时，使用这个 skill：
- 做竞品分析
- 比较几个产品
- 梳理某个赛道的市场格局
- 从产品经理角度拆解竞争策略
- 输出竞品对比结论和建议

这个 skill 的目标不只是收集信息，而是帮助用户把竞品研究转化为产品判断和下一步动作建议。

## 什么时候优先使用

以下表达通常都应视为适合触发：
- 竞品分析
- 产品对比
- 市场格局
- 竞品拆解
- 功能对标
- 机会点分析
- 差异化建议
- 帮我看看别人怎么做
- 这几个产品差异在哪
- 用户为什么选 A 不选 B

如果用户只是想了解单个公司简介、行业新闻或纯技术实现细节，而不是想比较、判断和提出建议，则不要使用这个 skill。

## 先判断赛道是否过大

如果用户只给了一个非常宽泛的赛道，不要直接开始竞品分析。

常见过大表达：
- 医疗赛道
- 教育赛道
- 电商赛道
- 金融赛道
- AI 赛道

这类表达通常无法直接分析，因为不同细分市场的目标用户、关键竞争要素、商业模式和主要竞品差异很大。

遇到这种情况时，应先做赛道收敛：
- 先指出当前赛道过大，直接分析容易失焦
- 帮用户拆成几个合理的细分方向
- 优先引导用户选择细分赛道
- 如果用户没有给目标用户，先基于赛道给出目标用户候选
- 如果用户没有给分析目标，先列出常见分析目标供其选择或补充
- 不要立刻要求用户自己定义核心场景
- 如果用户没有选择，但上下文已经明显指向某个方向，可以做出谨慎假设并明确说明

如果需要示例拆分方式，读取 [references/sector-scoping.md](references/sector-scoping.md)。

## 什么样的输出才算好

一份好的竞品分析，应该回答这些问题：
- 为什么要做这次分析
- 真正需要关注的竞品是谁
- 这个市场真正比拼的核心竞争要素是什么
- 这些竞品之间的关键差异是什么
- 用户为什么会选择 A 而不是 B
- 有哪些数据和证据支持这些判断
- 对用户当前的问题，下一步应该怎么做

除非用户明确要求，否则不要默认输出冗长的功能清单或只有纯文本的长段总结。

## 第一步：收敛赛道范围

先判断用户给出的赛道是否足够具体。

如果赛道已经足够清晰，例如“AI 病历生成”“互联网问诊”“医院信息化中的临床决策支持”，可以直接进入下一步。

如果赛道过大，例如“医疗”“教育”“电商”，应先拆成 3-6 个细分方向，并优先引导用户选择。

推荐收敛方式：
- 按目标用户拆分
- 按核心场景拆分
- 按产品形态拆分
- 按购买方或付费方拆分

在收敛完成前，不要急着给出竞品名单和结论。

## 第二步：生成目标用户候选并引导用户选择

如果用户没有明确目标用户，不建议让用户从零输入。

更好的做法是：
- 由模型基于细分赛道，先生成 3-5 个最相关的目标用户候选
- 用一句话说明每类目标用户为什么值得分析
- 引导用户选择 1-2 类最相关目标用户
- 如果用户不选，但上下文已经明显偏向某类目标用户，可以说明假设后继续

例如：
- 细分赛道：患者健康管理

模型可优先给出的目标用户候选：
- 患者
- 医生
- 医院管理者
- 保险或支付方

推荐原则：
- 优先给真正会使用、采购或强影响产品方向的人群
- 如果使用者和付费者不同，应优先同时提示
- 不要一次给太多选项，通常 3-5 个最合适

如果需要示例和规则，读取 [references/user-and-goal-selection.md](references/user-and-goal-selection.md)。

## 第三步：生成分析目标候选并引导用户选择

如果用户没有明确分析目标，不要直接假设“就是想看竞品”。

更好的做法是：
- 由模型列出常见分析目标候选
- 让用户选择最接近的一项，或自行补充
- 如果上下文已经明显偏向某个目标，可说明假设后继续

常见分析目标候选：
- 新产品立项
- 功能规划
- 定价或商业模式判断
- 增长策略参考
- 管理层或汇报材料

推荐原则：
- 让目标候选尽量互斥，避免含义重叠
- 优先使用用户最容易理解的表达
- 如果用户已经明确目标，不重复询问

如果需要示例和规则，读取 [references/user-and-goal-selection.md](references/user-and-goal-selection.md)。

## 第四步：生成核心场景候选并引导用户选择

当细分赛道、目标用户、分析目标已经基本明确后，不建议让用户从零输入核心场景。

更好的做法是：
- 由模型根据细分赛道、目标用户、分析目标，先生成 3-5 个最合理的核心场景候选
- 用简短说明告诉用户每个场景分别对应什么分析重点
- 引导用户选择 1-2 个最相关的核心场景
- 如果用户不选，但上下文已经明显偏向某个场景，可以说明假设后继续

例如：
- 细分赛道：患者健康管理
- 目标用户：医生、医院管理者
- 分析目标：新产品立项

模型可优先给出的核心场景候选：
- 慢病管理
- 院后随访
- 专病患者管理
- 健康干预与依从性管理

推荐原则：
- 候选场景应尽量互斥，避免重复表达
- 优先给高频、高价值、可形成差异化的场景
- 不要一次给太多选项，通常 3-5 个最合适

如果需要示例和规则，读取 [references/core-scenario-selection.md](references/core-scenario-selection.md)。

按目标选择重点时，读取 [references/analysis-goals.md](references/analysis-goals.md)。

## 第五步：定义竞品范围

在正式比较之前，先给竞品分组。

可根据需要使用以下分组：
- 直接竞品
- 替代方案
- 标杆参考产品

不要把所有产品都当作同一类对象来比较。如果有必要，说明为什么某个产品应归入该组。

## 第六步：识别市场竞争要素并选择分析维度

先识别该赛道最核心的市场竞争要素，再根据用户目标和已选核心场景选择分析维度。

市场竞争要素不是普通功能点，而是决定用户为什么选择某家产品、以及各家如何取胜的关键因子。

例如：
- 电商可重点看：多、快、好、省
- 教育可重点看：学习效果、价格、流量渠道、运营效率
- 工具类产品可重点看：效率、易用性、协作性、集成能力、价格

在未识别竞争要素之前，不要急着做泛化对比。

常见维度包括：
- 目标用户
- 核心使用场景
- 关键任务流程
- 功能广度与深度
- 体验亮点与短板
- 定价与商业模式
- 获客与增长方式
- 品牌与市场定位
- 战略方向

不要机械地把所有维度都塞进每一次分析。优先选择最有帮助的一组维度。

需要扩展维度时，读取 [references/dimension-library.md](references/dimension-library.md)。
需要判断赛道关键竞争要素时，读取 [references/competition-factors.md](references/competition-factors.md)。

## 第七步：绘制市场竞争格局

在开始逐个拆竞品之前，先把整个市场的竞争格局画出来。

这一部分不要只写“市场上有谁”，更要回答：
- 这个市场是按什么维度竞争的
- 不同企业、产品或业务形态分别占据什么位置
- 谁是头部，谁是中部，谁是长尾
- 哪些是直接竞品，哪些是替代方案
- 哪些位置已经拥挤，哪些位置仍有空位

默认建议至少包含：
- 一个二维四象限定位图
- 一个市场份额或市场分层判断

二维四象限的两个维度，不应随意拍脑袋决定。

默认规则是：
- x 轴和 y 轴都必须从该赛道已经识别出的“市场关键竞争要素”里选择
- 优先选最能把主要玩家拉开的 2 个要素
- 不同产品、企业或业务形态，适合观察的两个要素可以不同，但必须仍然来自同一套竞争要素库
- 如果竞争要素有 4 个以上，不要全塞进图里，而是选其中最有区分度的 2 个
- 需要明确说明：为什么选这两个，而不是另外两个

例如：
- 电商行业常见竞争要素是“多、快、好、省”
- 如果研究综合电商与即时零售，可优先选“多 x 快”
- 如果研究品牌电商与高客单零售，可优先选“好 x 省”或“好 x 多”
- 也就是说，轴的选择要跟分析对象和分析目标有关，但来源必须是市场竞争要素本身

例如：
- 电商可选：物品丰富度 x 送达时间
- 企业服务可选：标准化程度 x 行业深度
- 工具产品可选：单点效率 x 协作闭环能力

四象限中放的不一定都必须是单个产品，也可以是：
- 企业
- 产品
- 业务线
- 解决方案形态
- 替代方案

如果公开数据足够，补充：
- 市场份额分布图
- 头部、中部、长尾分层
- 主要玩家类型分布
- 市场类型判断，例如垄断、寡头、分散或长尾分散型

如果没有统一份额口径，也应至少给出：
- 分层判断
- 趋势判断
- 可替代指标，比如下载量、覆盖客户数、营收规模、医院覆盖数

需要系统分析市场格局时，读取 [references/market-landscape.md](references/market-landscape.md)。

## 第八步：收集关键数据与证据

在正式写竞品卡片之前，先收集能支撑判断的关键数据与证据。

优先关注：
- 市场份额
- MAU、DAU
- 付费用户量
- 注册用户量
- 下载量
- 营收或交易规模
- 用户增长趋势
- 官方披露或权威机构数据

如果需要移动应用相关数据，可优先参考七麦数据；如果需要行业规模或市场份额，可优先参考行业报告、财报、招股书、公司官方披露和可信媒体。

必须说明：
- 数据来源
- 数据口径
- 数据时间
- 是否为估算值

如果证据不足，不要硬凑数字，要明确标记“缺少公开数据”。

数据来源和使用原则，读取 [references/data-evidence.md](references/data-evidence.md)。

## 第九步：为每个核心竞品建立统一卡片

对于每个重要竞品，尽量用统一结构总结：
- 产品定位
- 所属企业或业务归属
- 目标用户
- 核心场景
- 它具体做什么
- 目标用户为什么会选它
- 它解决了什么痛点
- 是否刚需
- 所在赛道的关键竞争要素
- 主要能力
- 关键流程特征
- 核心壁垒
- 为什么只有它更有机会做成
- 业务核心逻辑
- 商业模式
- 盈利模式
- 增长闭环或飞轮
- 自生型 / 内耗型判断
- 盈利模式天花板或风险
- 收入结构
- 获客方式
- 留存方式
- 转化或商业化方式
- 交付方式
- 变现方式
- 关键数据指标
- 企业在整体业务中的角色
- 企业/产品关键路径的整体判断
- 关键阶段拆解
- 企业重要决策或战略动作
- 重要转折点
- 产品核心迭代路径
- 主要优势
- 主要短板
- 值得注意的战略信号

其中“企业/产品关键路径”不要只写成时间流水账。

更好的写法是按阶段拆解，每个阶段尽量说明：
- 这一阶段的主题是什么
- 为什么当时做这个决定
- 暴露了什么业务逻辑
- 如何改变了产品形态、用户结构或商业模式
- 对下一步方向意味着什么

如果竞品属于企业服务、平台型产品或解决方案型业务，还要补清楚：
- 它的购买方是谁
- 使用方是谁
- 它靠什么进入客户
- 它如何获客、留存、扩单和续费

如果公开信息足够，优先将关键阶段写成“阶段化路径拆解 + 时间轴”，而不是只列年份节点。

每个竞品卡片应尽量简洁，方便横向比较。

需要拆企业/产品关键路径时，读取 [references/company-product-path.md](references/company-product-path.md)。
需要拆核心壁垒、盈利模式和收入结构时，读取 [references/moat-business-revenue.md](references/moat-business-revenue.md)。

## 第十步：进行横向比较

不要停留在描述层面，要识别真正有意义的差异。

重点回答这类问题：
- 谁在关键竞争要素上更强
- 谁更适合哪类用户
- 谁更偏广覆盖，谁更偏深能力
- 谁主要依靠效率、体验、品牌、生态或价格竞争
- 谁的企业战略正在改变产品方向
- 哪些产品正在趋同
- 哪些产品形成了清晰差异化

相比“观察罗列”，更要输出“判断”。

## 第十一步：提炼产品经理视角的洞察

尽量始终产出以下几类洞察：
- 市场格局判断
- 市场关键竞争要素判断
- 核心竞品之间的策略差异
- 企业重大决策如何影响产品路线
- 用户做选择时的关键驱动因素
- 哪些壁垒是真壁垒，哪些只是阶段性优势
- 哪些盈利模式具备增长飞轮，哪些更偏内耗型
- 收入结构反映了企业正在押注什么
- 可切入的机会点或空白点
- 盲目模仿某个竞品的风险

如果证据不足，要明确说明，并保持结论克制。

## 第十二步：输出可执行建议

结尾一定要回到用户目标，给出建议。

常见建议方向包括：
- 哪些值得跟进
- 哪些不值得照搬
- 哪些地方适合做差异化
- 哪类用户或场景更值得优先切入
- 下一步最值得验证什么

建议必须具体，不能停留在空泛表态。

## 默认输出结构

除非用户要求其他格式，否则默认按以下结构输出：

1. 分析目标
2. 赛道范围定义
3. 目标市场与用户画像
4. 核心场景选择
5. 竞品分组
6. 市场关键竞争要素
7. 市场竞争格局
8. 核心竞品拆解
9. 横向比较
10. 行业发展路径
11. 关键洞察
12. 行动建议

需要按场景切换输出形式时，读取 [references/output-templates.md](references/output-templates.md)。

## 输出形式要求

默认优先输出结构化内容；如果用户要汇报、展示、沉淀材料或本地预览，优先输出为 HTML 报告，而不是纯文本。

HTML 报告应尽量使用适合的展示组件：
- 市场格局：分层图、定位图、摘要卡片
- 市场竞争格局：二维四象限、市场份额分布、玩家分层图
- 竞品对比：对比表、能力矩阵、差异卡片
- 行业发展路径：时间轴
- 企业/产品关键路径：阶段卡片 + 时间轴
- 数据佐证：指标表格、趋势块、来源标注
- 建议结论：结论卡片、优先级列表

HTML 结构和字段建议，读取 [references/html-report-spec.md](references/html-report-spec.md)。

如果已有结构化 JSON 数据，可使用 [scripts/render_html_report.py](scripts/render_html_report.py) 生成本地可预览的 HTML 文件。
如果需要从零开始组织输入数据，可参考 [assets/report-input-template.json](assets/report-input-template.json)。
如果想看一版完整示例，可参考 [assets/sample-ai-notes-analysis.json](assets/sample-ai-notes-analysis.json)。

## 工作风格要求

使用这个 skill 时，要遵循这些原则：
- 站在产品经理视角思考，而不是泛泛研究员视角
- 先判断赛道是否足够具体，再进入竞品分析
- 如果用户没有目标用户，优先由模型生成候选，再引导用户选择
- 如果用户没有分析目标，优先由模型生成候选，再引导用户选择或补充
- 核心场景优先由模型生成候选，再引导用户选择，而不是让用户从零定义
- 先识别赛道竞争逻辑，再判断产品差异
- 用数据和证据支撑关键判断
- 结合企业战略理解产品演化，而不是只看当前页面
- 将企业/产品关键路径写成“阶段 + 决策逻辑 + 影响 + 推演”，而不是公司大事记
- 将产品设计选择和用户价值、商业意图联系起来
- 避免空洞术语和套话
- 不要只停留在功能对比表
- 当信息不完整时，要明确假设
- 最终一定要形成观点

## 信息不足时怎么处理

如果用户没有提供竞品名单，请先提出一个合理的初始竞品集合。
如果用户只给了过大的赛道名称，应优先要求或帮助其收敛到细分赛道。
如果用户没有说明目标用户，应先基于细分赛道生成候选目标用户，再引导其选择。
如果用户没有说明分析目标，应先列出常见分析目标候选，再引导其选择或补充。
如果用户没有明确核心场景，应先由模型基于细分赛道、目标用户、分析目标给出候选核心场景，再引导选择。
如果用户没有说明输出形式，默认可先给结构化文本，再在适合时补 HTML 报告。
如果可获得信息较弱，请标记假设，不要过度下结论，并明确哪些结论缺少数据佐证。

需要判断哪些隐性请求适合触发时，读取 [references/example-scenarios.md](references/example-scenarios.md)。

## 不适用场景

以下情况不建议使用这个 skill：
- 单纯的行业新闻汇总
- 没有产品语境的纯技术代码对比
- 学术论文综述
- 没有比较目标的泛公司介绍
